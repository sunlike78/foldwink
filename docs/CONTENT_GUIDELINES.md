# Content Guidelines

## Content philosophy

The game will live or die on content quality.

A mediocre UI can survive.
Unfair puzzles will destroy trust immediately.

## What makes a good puzzle

A good puzzle is:

- understandable
- fair
- slightly deceptive
- satisfying once solved

A good puzzle is not:

- arbitrary
- based on trivia nobody knows
- technically valid but emotionally annoying
- dependent on niche culture unless intentionally themed

## Difficulty policy

### Easy

- broad familiar categories
- low ambiguity
- minimal cross-pollination

### Medium

- stronger false trails
- more abstract category relationships
- still broadly understandable

### Hard

Not for MVP.

## Vocabulary rules

Prefer:

- common nouns
- familiar verbs if clearly grouped
- everyday concepts
- themes with broad comprehension

Avoid:

- extremely regional slang
- spelling traps as primary difficulty
- brand-heavy groups unless themed
- ultra-obscure references
- words whose ambiguity is accidental rather than designed

## False trail design

### Good false trail

A pattern that tempts the player but collapses under inspection.

### Bad false trail

A pattern that is as plausible as the actual answer.

A false trail should create tension, not invalidate fairness.

## Content pipeline

1. generate draft
2. validate structure
3. detect duplicates
4. run ambiguity review
5. editorial pass
6. add to pool

No exceptions.

## LLM usage rules

LLM can help draft puzzles, but must not be trusted blindly.

Required when using LLM:

- ask for rationale per group
- ask for likely false trails
- ask for ambiguity risks
- validate with script
- manually review before publish

## Suggested content quotas for MVP

- total puzzles: 30 minimum
- easy: 18
- medium: 12
